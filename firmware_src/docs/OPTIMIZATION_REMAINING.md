## 🔴 核心功能缺失 (P0 - 必须立即处理)

### P0-01: MIDI 数据路由层 ⭐⭐⭐

**问题描述**：
- USB MIDI 数据未真正转发到网络
- 缺少路由规则和端口管理
- `app_core.c` 中只实现了基础转发

**影响范围**：核心功能不可用

**优化目标**：
- 创建 `midi_router` 组件
- 实现端口注册和路由规则
- 支持多种路由策略（广播、单播、组播）
- 打通 USB ↔ Network 双向数据通路

**预估工作量**：3-5 天

**验收标准**：
- [ ] USB MIDI 键盘数据能通过网络发送到 MIDI 2.0 设备
- [ ] 网络 MIDI 数据能发送到 USB MIDI 设备
- [ ] 支持基本的路由规则配置
- [ ] 支持端口映射和过滤

---

### P0-02: USB MIDI Host 功能完善 ⭐⭐⭐

**问题描述**：
- 当前实现是简化版本，缺少核心功能
- 设备枚举未实现（只有占位符）
- 事件回调为空
- 无实际 MIDI 数据接收逻辑

**影响范围**：USB MIDI 功能不可用

**优化目标**：
- 实现完整的 USB 设备枚举
- 解析 MIDI 类描述符
- 实现端点数据传输
- 处理热插拔事件
- 支持多个 USB MIDI 设备

**预估工作量**：5-7 天

**验收标准**：
- [ ] 能识别并连接 USB MIDI 键盘
- [ ] 能正确接收 MIDI 数据并触发回调
- [ ] 支持设备热插拔
- [ ] 支持最多 4 个 USB MIDI 设备

---

## 🟡 功能增强 (P2 - 提升用户体验)

### P2-01: 配置持久化 ⭐⭐

**问题描述**：
- WiFi 和 MIDI 配置无法保存，重启后丢失
- 每次都需要重新配置

**优化目标**：
- 创建 `config_manager` 组件
- 使用 NVS 存储配置
- 支持配置导入/导出
- 支持恢复出厂设置

**预估工作量**：2-3 天

**验收标准**：
- [ ] WiFi 凭证可持久化保存
- [ ] 设备名称、端口等配置可保存
- [ ] 支持恢复出厂设置
- [ ] 配置变更自动保存

---

### P2-02: 双向数据流（Network → USB）⭐⭐

**问题描述**：
- 当前只支持 USB → Network 单向
- 网络接收的 MIDI 数据无法发送到 USB

**优化目标**：
- 实现网络接收的 MIDI 数据转发到 USB
- 支持 USB MIDI Host 模式的发送功能
- 或切换到 USB MIDI Device 模式

**预估工作量**：3-4 天

**验收标准**：
- [ ] 网络 MIDI 数据能发送到 USB MIDI 设备
- [ ] 支持双向 MIDI 数据流
- [ ] 低延迟转发（<10ms）

---

### P2-03: WiFi Manager 增强 ⭐⭐

**问题描述**：
- 仅支持 STA 模式
- 无配网功能
- 断线重连逻辑简单

**优化目标**：
- 支持 Smart Config 或 SoftAP 配网
- WiFi 状态事件通知
- 断线自动重连优化
- WiFi 信号强度监控

**预估工作量**：2-3 天

**验收标准**：
- [ ] 支持首次配网流程
- [ ] WiFi 断线能自动恢复
- [ ] 提供 WiFi 状态 API
- [ ] 支持多 AP 配置

---

### P2-04: OTA 固件更新 ⭐

**问题描述**：
- 无远程更新能力
- 固件更新需要物理连接

**优化目标**：
- 配置 OTA 分区
- 实现基础 OTA 更新功能
- 支持回滚保护
- 提供更新状态 API

**预估工作量**：2-3 天

**验收标准**：
- [ ] 可通过 Web 或 API 触发 OTA
- [ ] 更新失败能自动回滚
- [ ] 支持版本验证
- [ ] 提供更新进度通知
